# 模型监控 vs 应用监控

阿里云百炼平台提供了两套互补的监控体系：**模型监控**和**应用监控**（应用观测）。模型监控聚焦于单个模型的调用性能、Token 消耗和用量统计；应用监控则面向完整的应用调用链路，追踪从 Prompt 输入到最终输出的端到端处理流程。两者的监控粒度、适用对象和使用场景各有不同，开发者需要根据实际需求选择合适的监控方案，或结合使用以获得全面的可观测性。

## 关键维度对比

| 维度 | 模型监控 | 应用监控（应用观测） |
|------|---------|-------------------|
| 监控粒度 | 单个模型级别 | 应用级别（端到端链路） |
| 监控对象 | 模型列表中的所有模型及自定义模型 | 智能体应用、工作流应用、高代码应用 |
| 核心指标 | 调用次数、Token 消耗、调用时长、首 Token 延时、RPM、TPM、失败率 | 调用次数、失败率、Token 总量、平均单次请求 Token 量、平均首 Token 耗时、平均调用时长 |
| 链路追踪 | 不支持（仅单模型维度） | 支持完整调用链路（LLM、检索、Embedding、重排序等多节点） |
| 调用日志 | 支持查看每次调用的输入/输出内容和 Token 消耗 | 支持查看 Prompt 内容、输出、延时和 Token 用量，可按 Request/Trace/Span ID 搜索 |
| 告警能力 | 支持（短信、邮件、电话、钉钉、企业微信、Webhook） | 不支持 |
| 外部集成 | 支持 Prometheus HTTP API，可接入 Grafana | 不支持（仅控制台操作） |
| 数据导出 | 不支持直接导出 | 支持导出为 JSONL 或 EXCEL |
| 数据标注 | 不支持 | 支持自定义标签标注，可导入评测集 |
| 地域限制 | 高级监控：北京、新加坡、弗吉尼亚；告警：北京、新加坡；日志：北京、新加坡部分模型 | 无特别说明 |
| 底层技术 | 私有 Prometheus 实例 | 可观测链路 OpenTelemetry 服务 |
| 计费 | 功能免费（模型调用本身按量计费） | 功能免费，但 OpenTelemetry 存储需付费 |
| 数据延迟 | 用量统计约 1 小时；日志分钟级 | 分钟级 |
| API 支持 | 支持 Prometheus HTTP API 查询 | 暂无 API，仅控制台 |

## 适用场景

### 适合使用模型监控的场景

- **模型性能调优**：需要关注特定模型的首 Token 延时、调用时长等性能指标，针对性优化推理效率。
- **成本管控**：按模型维度统计 Token 消耗和用量，识别高消耗模型，优化 Prompt 或切换轻量模型以降低成本。
- **异常告警**：需要配置主动告警规则，在调用失败率突增或 Token 消耗异常时及时收到通知。
- **自建监控系统**：已有 Grafana 等可视化平台，希望通过 Prometheus API 将百炼指标接入统一监控体系。

### 适合使用应用监控的场景

- **应用调试与排障**：需要追踪一次应用调用的完整链路（向量检索 -> 重排序 -> 大模型推理），定位具体环节的性能瓶颈或错误。
- **RAG 流程优化**：通过链路追踪查看检索、Embedding、Reranker 等各环节的延时和效果，优化整体检索增强生成流程。
- **评测数据积累**：利用数据标注和评测集导入功能，将线上真实调用数据转化为评测基准，持续迭代应用质量。
- **多节点工作流分析**：工作流应用包含多种节点类型（API、分类器、脚本、条件分支等），需要逐节点分析执行情况。

## 技术选型建议

对于大多数生产环境，建议**同时启用两套监控**：

1. 用**模型监控**建立成本和性能基线，配置告警规则实现被动防御，并通过 Prometheus 接口对接已有运维体系。
2. 用**应用监控**进行日常调试和链路分析，在出现异常时快速定位问题环节，同时积累标注数据持续优化应用效果。

如果资源有限需要二选一：仅做模型调用（未构建应用）时选模型监控；已在百炼平台构建智能体或工作流应用时优先开启应用监控，其端到端链路追踪能力更适合复杂应用场景的排障需求。
