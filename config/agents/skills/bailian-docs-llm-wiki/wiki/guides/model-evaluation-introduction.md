# model evaluation introduction

模型评测是百炼平台提供的模型质量验证工具，帮助开发者客观、量化地评估大语言模型的表现。其核心思路是：给模型出一套测试题，通过自动或人工打分，生成评测报告，从而为模型选型、调优验证和持续监控提供数据支撑。详细的功能说明请参见 [模型评测](../../raw/model-user-guide/model-evaluation-introduction/model-evaluation-overview.md)。

## 评测类型

百炼平台提供两种评测方式：

- **自定义评测**：适用于业务场景评测，支持自定义评分标准（评测维度），灵活性高。支持评测数据集（系统实时推理后评分）和推理结果集（直接对已有输出评分）两种数据来源。
- **基线评测**：使用预置的行业标准数据集（C-Eval、MMLU、ARC、GSM8K、HellaSwag、BBH），快速验证模型通用能力。**仅支持调优后的模型**，不支持预置模型。

如果是首次使用或需要针对特定业务场景评测，推荐自定义评测；如果需要验证调优模型的通用能力提升，推荐基线评测。最佳实践是两者结合使用。

## 支持的模型

**自定义评测**支持以下预置模型及基于它们调优后的模型：

- 千问系列：QwQ、Max、Plus、Turbo、Coder、Math
- 千问开源版：Qwen3、Qwen2.5、Qwen2、Qwen1.5
- 通义法睿
- 第三方文本生成模型（abab6.5g、abab6.5t 等）

> **注意**：以上列表可能随时变化，请以 [创建评测任务](https://bailian.console.aliyun.com/?tab=model#/efm/model_evaluate/task/creation) 页面中"评测模型"下拉列表为准。

**基线评测**仅支持调优后部署的模型，不支持预置模型（如千问-Max）。

## 评测维度（评分标准）

自定义评测需要先创建评测维度，即定义如何给模型打分。平台提供 5 种评测维度类型：

| 类型 | 评分方式 | 适用场景 | 成本 |
|------|---------|---------|------|
| 大模型评估-数值型 | AI 裁判打分（1-5 分） | 问答质量、内容生成质量 | 中 |
| 规则评估-文本相似度 | 算法计算相似度 | 翻译、摘要、改写 | 低 |
| 大模型评估-分类型 | AI 裁判分类（Pass/Fail） | 内容安全、正确性判断 | 中 |
| 规则评估-字符串匹配 | 精确字符串匹配 | Function Calling、NL2SQL | 极低 |
| 人工评估-分类型 | 人工打标签 | 创意性、专业判断 | 高 |

一个评测任务可以组合多个评测维度。

## 评测数据

**自定义评测**需要准备 Excel 格式的数据，包含以下列：

- **评测数据集**：Prompt（问题）+ Completion（参考答案），系统会调用模型推理后评分
- **推理结果集**：Prompt + Completion + Output（模型已有输出），系统直接对 Output 评分，无需再推理

建议数据量：小规模验证 50-100 条，正式评测 200-500 条，全面评估 500+ 条。

**基线评测**无需准备数据，直接选择系统预置的标准数据集即可，涵盖学科（C-Eval、MMLU、ARC）、数学（GSM8K）和推理（HellaSwag、BBH）三大类别。

## 核心变量

在大模型评估类型的评测维度中，裁判模型通过以下三个变量获取评测上下文：

- `${prompt}`：用户的原始问题或指令
- `${output}`：被评测模型生成的回答
- `${completion}`：参考答案（有标准答案时使用）

这些变量可在评分器 Prompt 中引用，用于指导裁判模型如何评判回答质量。

## System Prompt 与评分器 Prompt 的区别

这两种 Prompt 容易混淆，但作用于不同环节：

- **System Prompt**：在创建评测任务时配置，作用于**被评测模型**，用于设定模型的角色定位（如"你是客服人员"）。大多数情况下可留空。
- **评分器 Prompt**：在创建评测维度时配置，作用于**裁判模型**，定义评分标准和打分规则。仅大模型评估类型需要配置。

简单类比：System Prompt 告诉"学生"以什么身份答题，评分器 Prompt 告诉"阅卷老师"如何批改试卷。更多细节参见 [模型评测](../../raw/model-user-guide/model-evaluation-introduction/model-evaluation-overview.md) 中的相关章节。

## 典型使用场景

1. **模型选型**：用相同的业务测试数据评测所有候选模型，对比得分和通过率，用数据驱动选型决策。
2. **调优验证**：分别评测调优前后的模型（使用与训练集不重叠的评测数据），量化调优效果。
3. **能力量化**：生成包含综合得分、通过率、分数分布的评测报告，为团队沟通和管理层汇报提供数据支撑。
4. **持续监控**：建立定期评测机制，使用相同数据集追踪模型表现趋势，及时发现性能下降。

## 评测流程

完整的评测流程分为四个阶段，具体操作步骤参见 [模型评测](../../raw/model-user-guide/model-evaluation-introduction/model-evaluation-overview.md)：

1. **准备工作**：创建评测维度（定义评分标准）、准备测试数据
2. **创建评测任务**：选择模型、数据集、评测维度，决定是否参与排行榜
3. **执行评测**：自动评测由系统完成，人工评测需手动标注
4. **查看结果**：查看综合得分、通过率、数据明细，基线评测还可查看分学科得分和能力雷达图

## 评测报告

评测完成后可查看以下指标：

| 指标 | 说明 |
|------|------|
| 综合得分 | 评测维度的平均分 |
| 通过率 | 达到通过阈值的样本占比 |
| 数据明细 | 每条样本的 Prompt、Completion、Output 及评分 |

基线评测还额外提供分学科得分明细、能力雷达图和行业模型对比。

## 排行榜

排行榜用于对比多个模型在同一评测维度上的表现。创建评测任务时可选择是否参与排行榜，适用于需要在团队内部横向对比多个模型表现的场景。
