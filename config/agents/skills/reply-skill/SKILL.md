# reply-skill

把客户或同事的模糊提问，转换成一段基于当前项目代码的简短中文回复。

## 核心原则

- 先读代码，再回答。不要凭记忆、接口名或文件名猜。
- 同时覆盖前端和后端。当前目录通常是前端仓库，例如 `/Users/xlsama/w/store-health-web`；后端通常在同级目录，例如 `/Users/xlsama/w/store-health-api`。
- 只分析，不改代码。即使确认是 bug，也只说明问题原因和怎么调整，不执行修复。
- 输出要能直接转发给客户或同事，少术语、短句、结论先行。
- 证据要来自代码路径、调用链、字段逻辑或接口逻辑，不要泛泛而谈。

## 工作流

1. 确认当前仓库和同级仓库：
   - 先看 `pwd`、`package.json`、`pnpm-workspace.yaml`、`src`/`app`/`routes` 等入口。
   - 如果当前目录名像 `*-web`，在父目录找对应的 `*-api`。
   - 如果当前目录名像 `*-api`，在父目录找对应的 `*-web`。
   - 如果名字不匹配，用 `find .. -maxdepth 1 -type d` 看同级候选仓库。
2. 理解问题：
   - 如果用户发截图，先读截图里的页面、按钮、文案、报错和数据状态。
   - 如果描述模糊，提炼成一句“问题是什么”；能从代码验证就继续，不能验证才说明“当前证据只能判断到这里”。
3. 追代码链路：
   - 前端：从页面、路由、组件、表单、按钮、请求 hooks、状态管理、字段展示规则开始查。
   - 后端：从 Hono 路由、RPC 类型、参数校验、service、Drizzle 查询、权限/租户/门店过滤开始查。
   - 用 `rg` 查页面文案、接口路径、字段名、枚举值、报错文案、按钮名和相关类型。
   - 查所有相关调用方，不只看第一个命中的文件。
4. 判断问题类型：
   - bug/异常：说明触发条件、根因、建议怎么调整；不要改代码。
   - 需求/咨询/规则确认：直接回答当前代码支持什么、不支持什么、依据是什么。
   - 信息不足：给出基于代码的最小结论，并点明还缺哪一项输入才能继续确认。
5. 写最终回复前，删掉过程性细节，只保留客户需要知道的结论和必要证据。

## 输出格式

Bug 或异常类问题使用：

```text
问题是什么：一句话概括现象和影响。

问题原因：一句话概括根因；顺带说明建议怎么调整。

证据：
1. `<排查项>`：`<查了哪里>` 里 `<关键逻辑>`，所以 `<支撑的结论>`。
```

需求、咨询、规则确认类问题使用：

```text
问题是什么：一句话概括对方在问什么。

答案：一句话直接回答，不写“根因”。

证据：
1. `<排查项>`：`<查了哪里>` 里 `<关键逻辑>`，所以 `<支撑的答案>`。
```

信息不足时使用：

```text
问题是什么：一句话概括目前能确认的问题。

答案：当前代码只能确认到什么，还缺什么信息。

证据：
1. `<已确认的排查项>`：`<查了哪里>` 里 `<关键逻辑>`，所以目前只能确认 `<结论>`；还缺 `<信息>` 才能继续判断。
```

## 证据写法

- 证据不是固定数量。只列能支撑结论的排查项，一条足够就不要凑数。
- 每条证据写成“查了哪里 -> 关键逻辑是什么 -> 因此说明什么”。
- 可以写文件路径、函数名、组件名和接口名；除非用户要行号，否则不必列很多行号。
- 证据按链路顺序写：页面入口 -> 请求/状态 -> 后端接口/查询 -> 结论。
- 关键代码片段可以贴在对应证据项后面，但只贴最小片段；不要粘大段代码，不展开完整实现。
- 如果前后端只有一边相关，也要说明另一边查过但没有参与该逻辑。

代码片段格式：

````text
关键代码：
```ts
// 只贴最小关键片段
```
````

## 不做的事

- 不修改文件、不运行迁移、不提交代码。
- 不输出“我查看了很多文件”这类过程话术。
- 不写长篇排查报告。
- 不把内部思考、搜索过程或无关文件列给客户。
