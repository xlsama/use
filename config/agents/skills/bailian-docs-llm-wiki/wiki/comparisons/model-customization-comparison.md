# 模型微调、压缩与部署对比

百炼平台为开发者提供了从模型微调（Fine-Tuning）到模型压缩（Quantization）再到模型部署（Deployment）的完整模型定制化链路。三者分别解决"让模型更懂业务"、"让模型更省资源"和"让模型上线服务"三个阶段性问题。本文从核心目标、操作方式、支持模型、计费逻辑等维度进行对比，帮助开发者在不同阶段做出技术选型。

## 关键维度对比

| 维度 | 模型微调 | 模型压缩 | 模型部署 |
|------|---------|---------|---------|
| **核心目标** | 提升模型在特定业务场景下的表现 | 降低模型显存占用与部署成本 | 将模型上线为可调用的推理服务 |
| **输入** | 训练数据集（JSONL/XLSX/ZIP） | 全精度自定义微调模型 | 预置模型或自定义微调/压缩后模型 |
| **输出** | 自定义微调模型 | 量化后的低精度模型 | 在线推理服务（API 端点） |
| **操作方式** | 控制台 + API 双通道 | 仅控制台（API 支持校准数据传入） | 控制台 + API 双通道 |
| **支持模型范围** | Qwen3/2.5 系列、千问VL、万相图像/视频、CosyVoice 等 | 当前仅 qwen3.5-flash-2026-02-23 微调模型 | Qwen、DeepSeek、GLM、MiniMax、Kimi 等多系列 |
| **训练/处理方式** | CPT、SFT（全参/高效）、DPO（全参/高效） | 量化模板（指定目标部署规格） | PTU / 模型单元 / Token 用量三种计费模式 |
| **数据要求** | SFT: 1000+ 条问答对；CPT: 1000万+ Token；DPO: 100+ 组偏好对 | 部分量化模板需要校准数据集 | 无额外数据要求 |
| **计费方式** | 按 Token 计费（训练单元） | 当前限时免费 | PTU（按吞吐时长）/ 模型单元（按时长）/ Token 用量 |
| **可用地域** | 华北2（北京） | 华北2（北京） | 华北2（北京） |
| **可逆性** | 可反复训练新版本 | 不可逆，不支持二次压缩或继续训练 | 可随时上下线、扩缩容 |

## 流程关系

三者在模型定制化链路中存在上下游依赖关系：

```
模型微调 ──产出全精度模型──→ 模型压缩（可选）──产出量化模型──→ 模型部署
                              │                                    ↑
                              └────────────────直接部署─────────────┘
```

- **模型压缩**的输入必须是模型微调产出的全精度自定义模型，不能对预置模型或已量化模型执行压缩。
- **模型部署**可以直接部署微调后的全精度模型，也可以部署压缩后的量化模型，两者均支持。
- 预置模型无需微调或压缩，可直接进入部署环节。

## 适用场景建议

### 场景一：快速验证微调效果

完成微调后，使用 **Token 用量**计费方式直接部署微调模型进行效果验证，不使用不计费，适合低频测试。

### 场景二：生产环境高并发服务

微调模型效果达标后，选择 **PTU** 或**模型单元**计费方式进行部署。PTU 适合流量稳定的高吞吐场景，模型单元适合需要灵活调整推理参数的场景。

### 场景三：降低部署成本

当微调模型的部署规格过高（如需要 MU1 * 2），可先通过**模型压缩**将模型量化到更小规格（如 MU8 * 1），部署成本可降低约 56%，再进行部署。

### 场景四：多模态模型定制

对视觉理解、图像生成、视频生成、语音合成等多模态场景，通过微调注入风格或领域特征后直接部署，当前这些模型暂不支持压缩。

## 技术选型决策参考

| 决策点 | 推荐方案 |
|--------|---------|
| 模型效果不满足业务需求 | 先做模型微调（SFT 优先，必要时叠加 CPT/DPO） |
| 微调模型部署成本过高 | 评估模型压缩（当前仅部分模型支持） |
| 需要稳定高吞吐推理服务 | 选择 PTU 计费部署 |
| 需要灵活控制推理参数（Thinking 模式、上下文长度等） | 选择模型单元计费部署 |
| 低频调用或效果验证 | 选择 Token 用量计费部署 |
| 已有本地 LoRA 权重 | 通过 OSS 导入后直接部署 |
