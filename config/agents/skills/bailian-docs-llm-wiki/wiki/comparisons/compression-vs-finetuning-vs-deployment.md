# 模型压缩 vs 微调 vs 部署

百炼平台围绕自定义模型提供了三项核心能力：**模型压缩**（量化降本）、**模型微调**（Fine-Tuning，效果提升）和**模型部署**（上线推理服务）。三者分别解决"模型太大成本高"、"模型效果不够好"和"模型如何对外提供服务"这三个不同阶段的问题。本文从输入输出、支持模型、操作方式、计费等关键维度进行对比，帮助开发者快速理解各功能的定位与适用场景，做出合理的技术选型。

## 关键维度对比

| 维度 | 模型压缩 | 模型微调（Fine-Tuning） | 模型部署 |
|------|---------|----------------------|---------|
| **核心目标** | 降低推理显存占用与部署成本，提升推理吞吐 | 提升模型在特定业务场景下的表现 | 将模型上线为可调用的推理服务 |
| **输入** | 全精度自定义微调模型 + 可选校准数据集 | 标注好的训练数据集（.jsonl / .xlsx） | 预置模型或调优/压缩后的模型 |
| **输出** | 低精度量化模型（可直接部署） | 自定义微调模型（可部署或进一步压缩） | 独享推理服务（返回 deployed_model ID） |
| **支持的模型类型** | 仅支持特定基础模型的自定义微调模型（当前支持 qwen3.5-flash-2026-02-23） | 文本生成、视觉理解、图像生成、视频生成、语音合成等多种模型 | 千问系列、DeepSeek 系列、GLM、MiniMax、Kimi、CosyVoice 等预置模型及调优后模型 |
| **操作方式** | 控制台创建压缩任务 | 控制台可视化操作 或 DashScope API | 控制台部署 或 HTTP API |
| **API 端点** | 控制台操作为主；OpenAPI 支持 `custom_calibration_file_ids` 参数 | `POST /api/v1/files`（上传）、`POST /api/v1/fine-tunes`（创建任务）、`GET /api/v1/fine-tunes/<job_id>`（查询） | `POST /api/v1/deployments`（创建）、`GET /api/v1/deployments/{id}`（查询）、`DELETE /api/v1/deployments/{id}`（删除） |
| **计费方式** | 压缩过程当前限时免费；压缩后模型按部署规格计费 | 按 Token 计费，使用训练单元 | 三种方式：预置吞吐（PTU）、模型单元（MU）、Token 用量 |
| **可逆性** | 不可逆，产出模型不支持继续训练或二次压缩 | 可多次迭代训练，支持在不同数据集上反复调优 | 可随时上线/下线，支持扩缩容 |
| **地域限制** | 以控制台为准 | 仅华北2（北京） | 仅华北2（北京） |

## 流程关系对比

三个功能在模型生命周期中处于不同阶段，典型的串联流程为：

| 阶段 | 功能 | 说明 |
|------|------|------|
| 第一步 | 模型微调 | 基于业务数据训练，产出全精度自定义模型 |
| 第二步（可选） | 模型压缩 | 将全精度模型量化，降低部署规格和成本 |
| 第三步 | 模型部署 | 将微调或压缩后的模型上线为推理服务 |

## 计费方式详细对比

| 计费维度 | 模型压缩 | 模型微调 | 模型部署 |
|---------|---------|---------|---------|
| **计费单位** | 当前免费 | 训练单元（按 Token） | PTU：按吞吐量和时长；MU：按模型单元数和时长；Token：按用量 |
| **付费模式** | - | 按量后付费 | PTU/MU 支持后付费（按小时）和预付费；Token 按量 |
| **成本示例** | 压缩 qwen3.5-flash 微调模型后，部署从 MU1x2（108元/时）降至 MU8x1（47元/时） | 根据数据量和训练轮次按 Token 计费 | 根据所选计费方式和规格不同而异 |

## 适用场景建议

### 选择模型微调的场景

- 模型在目标任务上的表现不够理想，需要注入领域知识或学习特定指令格式
- 拥有高质量的标注数据（SFT 至少 1000 条问答对，CPT 至少 1000 万 Token）
- 需要模型对齐特定偏好（安全合规、输出风格等），可使用 DPO

### 选择模型压缩的场景

- 已经完成微调并验证效果，但部署成本过高
- 希望在不显著损失精度的前提下降低推理资源消耗
- 需要提升推理吞吐量以应对更高并发

### 选择模型部署的场景

- 需要将预置模型或自定义模型对外提供稳定的推理服务
- 对并发性能、延迟有明确要求（PTU 适合高负载生产环境，MU 适合需要灵活配置的场景）
- 仅需验证调优后模型效果（Token 用量计费，不使用不计费）

### 三者组合使用的典型路径

1. **效果优先**：先微调 → 验证效果 → 直接部署（跳过压缩）
2. **成本优先**：先微调 → 压缩降低规格 → 部署节省成本
3. **快速上线**：选择预置模型 → 直接部署（跳过微调和压缩）

## 技术选型决策参考

- **是否需要提升模型效果？** 是 → 先做模型微调；否 → 直接部署预置模型
- **微调后部署成本是否可接受？** 是 → 直接部署；否 → 先做模型压缩再部署
- **业务流量模式如何？** 稳定高并发 → PTU 部署；需灵活调整 → MU 部署；仅验证效果 → Token 用量部署
- **是否需要导入自训练模型？** 是 → 通过模型部署的 OSS 导入功能，支持 LoRA 模型导入后部署
