# 模型优化方式对比：微调、压缩与部署

百炼平台围绕自定义模型提供了三种互补的优化与落地能力：**模型微调（[fine tuning](../guides/fine-tuning.md)）** 负责让基础模型学到领域知识与任务能力，**模型压缩（[model compression](../guides/model-compression.md)）** 负责把全精度微调模型量化为低精度版本以降低部署成本，**模型部署（model deployment）** 负责把模型发布为资源专享的推理服务。三者构成「训练 → 压缩 → 部署」的递进链路，也各自可独立使用。本页从目标、输入要求、支持模型、操作方式、计费和典型场景等维度做横向对比，帮助开发者快速完成技术选型。

## 关键维度对比

| 维度 | 模型微调（Fine Tuning） | 模型压缩（Model Compression） | 模型部署（Model Deployment） |
| --- | --- | --- | --- |
| 核心目标 | 注入领域知识、对齐指令与偏好，产出自定义模型 | 将全精度微调模型量化为低精度版本，降显存与成本 | 把预置或调优后模型部署为专享推理服务 |
| 在链路中的位置 | 上游：产出全精度自定义模型 | 中游：对微调产出物做量化 | 下游：对外提供推理 API |
| 输入要求 | SFT 需 1000+ 条「问-答」对；DPO 需 100+ 组偏好对；CPT 需 1000 万+ Token 领域文本；图像/视频/CosyVoice 各有专用数据集格式 | 当前工作空间内存在支持压缩的自定义微调模型；部分量化模板需校准数据集 | 已有预置模型或调优/压缩后模型；API 部署需 API Key 并配置环境变量 |
| 支持模型 | Qwen 系列文本生成（CPT/SFT/DPO）、千问 VL（SFT）、万相图像/视频（SFT-LoRA）、CosyVoice（SFT 高效微调）等 | 当前支持 qwen3.5-flash-2026-02-23 等基础模型对应的自定义微调模型，具体以控制台为准 | 千问、DeepSeek、千问 VL、千问 Omni、GLM、CosyVoice 等系列，以及调优/压缩后模型 |
| 训练/压缩方式 | CPT、SFT、DPO 三种方式；全参训练与 LoRA 高效训练两种模式 | 量化模板（如 MU5、MU8），可选校准数据 | 预置吞吐（PTU）、模型单元（MU）、Token 用量三种计费方式 |
| 操作入口 | 控制台「模型调优」页面或 DashScope API / 命令行 | 控制台「模型 > 模型训练 > 模型压缩」页面 | 控制台「模型部署」或调用 `https://dashscope.aliyuncs.com/api/v1/deployments` |
| 输出产物 | 自定义微调模型（全参或 LoRA），可用于压缩或部署 | 量化后的低精度模型，进入「我的模型」可部署 | 专属推理服务（`deployed_model` ID），可发推理请求 |
| 计费方式 | 按 Token 用量计费（高效训练可冻结 ViT 后按 Token 计费） | 压缩功能当前限时免费；产出模型部署按部署单元标准规格计费 | PTU：时长 × (输入 TPM 单价 × 输入 TPM + 输出 TPM 单价 × 输出 TPM)；模型单元：时长 × 单元数 × 单价（支持包月）；Token 用量：输入/输出 Token 数 × 单价 |
| 可逆性 | 可迭代训练（CPT→SFT→DPO 组合） | 不可逆：产出模型不支持继续训练或二次压缩 | 部署后计费方式不可更改，切换需先下线再重新部署；删除服务立即停止计费 |
| 地域限制 | 仅适用于华北2（北京），需使用该地域 API Key | — | 仅适用于华北2（北京） |

## 各方案适用场景

### 模型微调

- 需要让模型掌握特定领域知识或遵循企业指令风格。
- 数据量充足（SFT 千条以上、DPO 百组以上、CPT 千万 Token 以上），希望进一步提升效果。
- 需要支持多模态（视觉理解、图像/视频生成、语音合成）场景的定制。
- 推荐流程：`CPT（可选）→ SFT → DPO（可选）`；数据集小或需快速验证时优先用 LoRA 高效训练。

### 模型压缩

- 已有全精度微调模型，但部署规格偏高、成本压力大。
- 对延迟和吞吐有要求，希望缩减部署单元规格。
- 典型案例：qwen3.5-flash-2026-02-23 微调模型压缩前 MU1 × 2（108 元/小时），压缩后 MU8 × 1（47 元/小时）。
- 注意：压缩不可逆，迭代需回到上游全精度模型重新训练后再压缩。

### 模型部署

- 高负载生产环境、需稳定 TPS 与低延迟：选 **PTU**，支持长输入与前缀缓存。
- 私有模型独占资源、长时任务、PD 分离模式：选 **模型单元**，可自定义吞吐/并发/上下文/限流。
- 调优后模型效果验证、追求性价比：选 **Token 用量**（仅支持 LoRA 调优后模型）。
- 从 OSS 导入本地训练的 LoRA 模型后，再部署为专享服务。

## 技术选型建议

1. **先训后压再部署**：若目标是把基础模型改造为低成本生产服务，按「微调 → 压缩 → 部署」顺序串联，能在保证效果的同时最小化部署成本。
2. **小数据快速验证**：数据有限或需快速迭代时，微调阶段用 LoRA 高效训练，部署阶段用 Token 用量计费，验证通过后再升级到 PTU 或模型单元。
3. **成本敏感型生产**：微调产出全精度模型后务必评估是否压缩；压缩后按更小部署单元规格上线，可显著降低长期运行费用。
4. **稳定高并发场景**：直接选用 PTU 部署预置模型或调优后模型，配合长输入与前缀缓存降低单位 token 成本；避免额度超限自动转按量计费。
5. **不可逆操作需谨慎**：压缩产出模型不可继续训练、不可二次压缩；部署计费方式创建后不可更改——在执行前务必确认配置。
