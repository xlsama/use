# 通用用例与应用用例对比

百炼平台的官方文档中存在两份"使用场景"指南：一份是面向**模型与算法侧**的通用用例（`use cases`），聚焦 Prompt 工程、多模态生成、RAG 管线、模型调优与平台限流等能力本身；另一份是面向**应用与集成侧**的应用用例（`application use cases`），聚焦如何把已发布的应用快速接入网站、微信、企业微信、钉钉等终端渠道。两者解决的问题不同、开发者角色不同、调用链路也不同，技术选型时常被混淆，本页通过逐维度对比帮助开发者快速定位该读哪一份、该走哪条路。

## 关键维度对比

| 维度 | 通用用例（[use cases](../guides/use-cases.md)） | 应用用例（[application use cases](../guides/application-use-cases.md)） |
|------|----------------------|-----------------------------------|
| 核心目标 | 发挥模型能力、调优生成质量、构建检索增强管线 | 把已建好的应用一键集成到外部渠道 |
| 面向角色 | 算法/数据工程师、Prompt 工程师 | 业务/应用开发、运营与集成工程师 |
| 入口对象 | DashScope 模型 API（qwen-max、万相文生图/视频、Vidu 等） | 百炼控制台智能体应用 + AppFlow 连接器 |
| 典型能力 | Prompt 工程（文生文/文生图/文生视频）、多模态、RAG、自定义模型调优、限流与缓存 | 渠道集成（网站/微信公众号/企业微信/钉钉）、本地 RAG、知识库挂载 |
| 输入形式 | API 调用：`prompt`、`negative_prompt`、训练数据（Prompt-Completion） | 控制台应用配置 + AppFlow 连接流 + 挂件脚本 |
| 输出形式 | 文本、图片、视频、微调后模型、评测报告 | 渠道内可交互的 AI 助手/客服机器人 |
| 模型使用 | 直接调用各类基础模型与第三方模型（DeepSeek、Kimi、GLM、MiniMax、MiMo、Stepfun、Vidu 等） | 主要通过应用层调用 Qwen3.5-Plus、千问-Plus/Turbo 等 |
| RAG 实现 | `LlamaIndex` + `DashScopeCloudIndex` 云端检索；或自定义切分与嵌入 | 控制台知识库（默认或 ADB-PG 向量库），应用中"必定调用"；或本地 Gradio + LlamaIndex |
| 集成方式 | SDK / API 调用，代码为主 | AppFlow 无代码连接器 + 悬浮挂件脚本，少量代码 |
| 部署形态 | 模型推理服务、自定义模型独占实例 | 渠道侧部署（网站挂件、公众号后台、企业微信回调、钉钉机器人） |
| 计费方式 | 按模型推理 token 计费；Prompt 自动优化工具按推理费用计费；自定义模型部署按独占实例规格计费 | 应用调用走模型推理计费；AppFlow 与部分渠道连接器可能产生额外连接费用 |
| 数据准备 | 训练集为 Prompt+Completion，评测集仅 Prompt；建议 ≥500 条；单次导入 ≤10 文件 | 知识库导入 pdf/doc/docx/txt/md，单文档 ≤100MB 或 1000 页 |
| 典型场景 | 文生图/视频创作、Prompt 调优、领域模型微调、检索增强问答原型 | 7x24 智能客服、公众号问答机器人、钉钉办公助手、网站悬浮助手 |
| 关键约束 | 限流按主账号 + 模型独立计算，触发后约 1 分钟恢复；微调模型必须部署后才能评测 | 公众号未认证消息响应 5 秒超时；企业微信需域名主体一致与企业可信 IP |

## 适用场景建议

### 选"通用用例"当你：

- 需要直接调用某个模型 API（文生文、文生图、文生视频、Vidu）并打磨 Prompt。
- 要基于 LlamaIndex 把百炼检索增强服务接入自有检索管线。
- 打算用自有领域数据微调出专属模型，并做评测迭代。
- 关注平台层面的限流、配额、缓存优化策略。

这类工作以"调模型 / 调参数 / 调数据"为主，产物是更好的生成结果或一个微调后的模型，几乎不涉及"把应用挂到某个 IM 上"。

### 选"应用用例"当你：

- 已在百炼控制台建好智能体应用，想 10 分钟内把它接入网站、微信公众号、企业微信或钉钉。
- 业务诉求是"一个能 7x24 回答用户问题的渠道机器人"，而非底层模型调优。
- 希望用 AppFlow 无代码方式串联应用与渠道，不愿写大量集成代码。
- 需要把企业私有文档挂到知识库，让助手在专业领域回答更准确。

这类工作以"配应用 / 配渠道 / 配知识库"为主，产物是一个在线上渠道里可交互的 AI 助手。

### 两者并用的场景

最常见的真实链路是**先用通用用例打磨模型与 Prompt，再用应用用例把成果发布到渠道**：例如先在通用用例侧用 LlamaIndex + qwen-max 验证 RAG 效果、调好 Prompt 与召回参数，再到应用用例侧创建智能体应用、挂载同一批知识库文档、通过 AppFlow 接入钉钉，形成"模型层 → 应用层 → 渠道层"的完整闭环。此时通用用例负责"质量上限"，应用用例负责"触达与运营"。

## 技术选型参考

- **只要模型 API，不要渠道**：直接走通用用例的 DashScope API，无需创建控制台应用，避免 AppFlow 依赖。
- **只要渠道机器人，不深入调优**：直接走应用用例，用控制台默认模型（Qwen3.5-Plus / 千问-Plus）+ 知识库即可，无需自建 RAG 管线。
- **需要私有化检索或自定义切分**：通用用例的 LlamaIndex 方案与本地 RAG 应用（[application use cases](../guides/application-use-cases.md) 中的本地 RAG）都能满足，前者偏云端托管、后者偏本地灵活，按是否需要本地嵌入模型选择。
- **需要领域专属模型**：只能走通用用例的自定义模型调优流程，应用用例不提供微调入口。
- **对响应时延敏感的渠道**（如未认证公众号 5 秒超时）：优先走应用用例，选千问-Turbo 并在 Prompt 中要求简短回答；通用用例侧的 Prompt 调优经验可复用过来。
