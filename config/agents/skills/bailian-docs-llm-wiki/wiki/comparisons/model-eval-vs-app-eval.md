# 模型评测 vs 应用评测

百炼平台提供两套独立的评测体系：**模型评测**和**应用评测**。前者聚焦于大语言模型本身的能力验证与选型，后者聚焦于基于模型构建的智能体/工作流应用的端到端输出质量。两者在评测对象、数据准备、评分机制和适用阶段上存在显著差异，理解它们的区别有助于开发者在不同阶段选择正确的评测手段。

## 关键维度对比

| 维度 | 模型评测 | 应用评测 |
|------|---------|---------|
| **评测对象** | 大语言模型（预置模型或调优后模型） | 智能体应用、工作流应用 |
| **核心目标** | 验证模型本身的语言能力与任务表现 | 验证应用的端到端输出质量，包括检索、重排、Prompt 等全链路 |
| **评测模式** | 自定义评测、基线评测 | 自动评测（单应用/多应用横评）、手动评测、新版评测任务 |
| **数据准备** | Excel 格式（Prompt + Completion），基线评测免准备 | 自动生成评测集（基于知识库）或手动上传（xls/xlsx/jsonl） |
| **评分方式** | AI 裁判打分、规则评估（相似度/字符串匹配）、人工评估 | 预置评估器模板、LLM 评估器、Code 评估器、人工标签标注 |
| **评估维度定义** | 评测维度（5 种类型，需手动创建） | 评估器 + 标签（预置模板或自定义，最多 10 个评估器） |
| **归因分析** | 无（仅给出得分和通过率） | 支持 BadCase 自动归因（模型理解、重排、检索、切片、未获取知识） |
| **调优建议** | 无（需开发者自行分析报告） | 自动生成针对 Prompt、检索配置、知识库切片的优化建议 |
| **支持模型范围** | 千问系列、通义法睿、部分第三方模型；基线评测仅限调优后模型 | 仅限已发布且配置知识库的智能体应用，或工作流应用 |
| **前置条件** | 无特殊要求 | 需开通应用观测，将应用添加到观测列表 |
| **多对象对比** | 通过排行榜对比多个模型在同一维度的表现 | 多应用横向评测（最多 8 个应用同时对比） |
| **报告内容** | 综合得分、通过率、数据明细；基线评测含雷达图 | 总正确率、BadCase 分析、归因分析、调优建议 |
| **计费** | 使用 AI 裁判打分时按 Token 计费，规则评估无额外费用 | 自动评测和 LLM 评估器按 Token 计费，Code 评估器无额外费用 |

## 适用场景建议

### 选择模型评测的场景

- **模型选型阶段**：需要在多个候选模型之间做横向对比，用相同测试数据评估各模型的基础能力。
- **模型调优验证**：完成模型微调后，需要量化调优前后的能力变化。
- **通用能力基准测试**：使用 C-Eval、MMLU 等标准数据集，快速检验调优模型在学科、数学、推理等通用维度的表现。
- **持续能力监控**：建立定期评测机制，追踪模型能力随时间的变化趋势。

### 选择应用评测的场景

- **应用上线前验证**：智能体或工作流应用发布前，端到端验证回答质量，定位 Prompt、检索、知识库等环节的问题。
- **知识库更新后回归**：知识库内容变更后，评估变更对应用回答质量的影响。
- **多应用方案选型**：在多个应用方案之间横向对比，选择最优配置。
- **线上质量持续运营**：结合应用观测数据，对线上真实流量进行标注和评测，发现并修复 BadCase。

### 两者结合使用

推荐的完整质量保障流程为：先用**模型评测**选定基础模型，再基于该模型构建应用，最后用**应用评测**验证应用的端到端效果。当应用评测发现问题归因到模型能力不足时，可回到模型评测环节重新选型或调优。

## 技术选型参考

对于刚接触百炼平台的开发者，可按以下决策路径选择评测方式：

1. 如果尚未确定使用哪个模型，先进行**模型评测**（自定义评测 + 基线评测），用数据驱动模型选型。
2. 如果模型已确定、正在构建应用，使用**应用评测**验证应用整体质量，利用归因分析快速定位问题环节。
3. 如果需要对调优模型做标准化验证，使用模型评测的**基线评测**，无需自行准备数据。
4. 如果需要人工深度分析应用回答质量，使用应用评测的**标签标注**功能，支持分类、布尔值、数字、文本四种标注类型。
