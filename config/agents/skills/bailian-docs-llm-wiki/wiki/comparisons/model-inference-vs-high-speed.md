# 标准推理 vs 高速推理

百炼平台为开发者提供了两种推理调用方式：**标准推理**（按量付费，使用公共资源池）和**高速推理**（TPM 预留，锁定专属容量）。两者在容量保障、计费模式和接入方式上存在显著差异。本文从多个维度进行对比，帮助开发者根据业务特征做出技术选型。

## 核心维度对比

| 维度 | 标准推理（按量付费） | 高速推理（TPM 预留） |
|------|---------------------|---------------------|
| 容量保障 | 无，共享公共资源池 | 专属容量，刚性兑付 |
| 计费方式 | 按实际消耗 token 计费 | 按 kTPM 预付费（按天） |
| 限流风险 | 高峰期可能受公共限流影响 | 预留额度内不受限流影响 |
| 超额处理 | 触发限流，请求可能被拒绝 | 超额部分自动降级为按量计费，服务不中断 |
| 接入方式 | 使用标准模型 ID 调用 | 需替换为专属模型 code |
| 代码改动 | 无需改动 | 需将 `model` 参数替换为专属模型 code |
| 支持模态 | 文本、视觉、音频、视频、3D 等全模态 | 目前主要支持文本生成模型 |
| 支持模型范围 | 全部百炼模型（含第三方） | 部分模型（qwen3 系列、deepseek、kimi 等） |
| 费用优化 | 可叠加资源包/节省计划 | 预留容量内不额外收费，超额按量计费 |
| 适合流量模式 | 波动大、难以预估 | 可预估、持续稳定 |

## 支持模型对比

| 模型 | 标准推理 | 高速推理（TPM 预留） |
|------|---------|---------------------|
| qwen3.7-max | 支持 | 支持 |
| qwen3.7-plus | 支持 | 支持 |
| qwen3.6-flash | 支持 | 支持 |
| deepseek-v4-pro | 支持 | 支持 |
| deepseek-v4-flash | 支持 | 支持 |
| kimi-k2.6 | 支持 | 支持 |
| 视觉/音频/视频模型 | 支持 | 暂不支持 |
| 第三方模型（glm、MiniMax 等） | 支持 | 部分支持 |

## 计费模式对比

| 项目 | 标准推理 | 高速推理 |
|------|---------|---------|
| 付费方式 | 后付费（按量） | 预付费（按 kTPM/天） |
| 最低承诺 | 无 | 需购买最低 kTPM 额度 |
| 缓存折扣 | 视模型而定 | 支持（Qwen 系列 0.2 折算，DeepSeek 0.08 折算） |
| 退费机制 | 不涉及 | 支持退订，按已用时长折算退款 |
| 扩缩容 | 不涉及 | 支持在线扩缩容，变配期间服务不中断 |

## 适用场景建议

### 选择标准推理的场景

- 业务处于早期探索阶段，流量不可预估。
- 需要调用多种模态（视觉、音频、视频等）的模型。
- 流量波动大，峰谷差异明显，不适合预留固定容量。
- 对偶尔的限流可以容忍，或已通过排队/重试机制应对。

### 选择高速推理的场景

- 业务流量可预估，且对限流零容忍（如线上客服、实时对话）。
- 主要使用文本生成模型，调用量集中在少数几个模型上。
- 需要稳定的推理延迟，不希望受公共资源池波动影响。
- 愿意通过预付费换取确定性容量保障。

## 技术选型参考

1. **先评估流量模式**：使用标准推理的监控数据观察 RPM 和 TPM 用量趋势，判断是否存在持续的高位调用。
2. **从标准推理起步**：新业务建议先使用标准推理验证模型效果，待流量稳定后再考虑迁移到 TPM 预留。
3. **混合策略**：核心业务模型使用 TPM 预留保障容量，非核心或低频模型继续使用标准推理按量付费。
4. **容量规划**：TPM 预留提供容量计算器，可根据每分钟请求数、平均输入输出长度和缓存命中率估算所需额度。
5. **超额降级兜底**：即使选择 TPM 预留，超额请求也会自动降级为按量计费，无需担心服务中断。
